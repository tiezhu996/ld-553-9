## 项目主要需求

### 项目概述

TransitHub 是一个交通出行管理平台，面向交通运营企业管理者、司机和运维人员，支持车辆管理、充电桩管理、出行订单处理、维修记录管理等核心业务。平台集成高德地图 API，提供地图总览、车辆实时定位、充电桩分布展示、出行轨迹回放等可视化能力。

系统采用前后端分离架构：前端 Angular 17 + TypeScript + Angular Material，后端 Django + Django REST Framework，数据库 PostgreSQL，JWT 进行身份认证，Redis 用于缓存和会话管理。

**特别注意**：本项目使用 Angular 框架（不是 React/Vue），技术栈与常见的 React/Vue 项目有显著差异。使用 Angular CLI 默认的 webpack 构建。高德地图 API 的 key 通过环境变量注入。

```python
# Django Model
class Vehicle(models.Model):
    id = models.AutoField(primary_key=True)
    plate_number = models.CharField(max_length=20, unique=True)  # 车牌号
    type = models.CharField(max_length=20, choices=VehicleType.choices)  # 公交/出租/网约车/物流
    brand = models.CharField(max_length=50)         # 品牌
    model = models.CharField(max_length=50)         # 型号
    year = models.IntegerField()                    # 出厂年份
    status = models.CharField(max_length=20, choices=VehicleStatus.choices)  # 状态
    driver = models.ForeignKey('User', on_delete=models.SET_NULL, null=True, blank=True)  # 司机
    insurance_expiry = models.DateField()           # 保险到期日
    created_at = models.DateTimeField(auto_now_add=True)
    updated_at = models.DateTimeField(auto_now=True)
```

- **VehicleStatus 状态流转**：
  ```
  PENDING（待审核） → OPERATING（运营中） → MAINTENANCE（维修中） → OPERATING
                                    ↘ DISABLED（停用/报废）
  ```
- **业务动作**：
  - 注册车辆（创建记录，状态 PENDING）
  - 审核车辆（PENDING → OPERATING）
  - 调度车辆（分配司机）
  - 维修车辆（OPERATING → MAINTENANCE）
  - 报废车辆（任何状态 → DISABLED）

```python
class ChargingPile(models.Model):
    id = models.AutoField(primary_key=True)
    code = models.CharField(max_length=50, unique=True)  # 充电桩编号
    location = models.CharField(max_length=200)           # 地址描述
    lat = models.DecimalField(max_digits=10, decimal_places=7)  # 纬度
    lng = models.DecimalField(max_digits=10, decimal_places=7)  # 经度
    type = models.CharField(max_length=10, choices=PileType.choices)  # 快充/慢充
    power = models.DecimalField(max_digits=8, decimal_places=2)  # 功率（kW）
    status = models.CharField(max_length=20, choices=PileStatus.choices)  # 状态
    price_per_kwh = models.DecimalField(max_digits=6, decimal_places=2)  # 每度电价格
    installed_at = models.DateTimeField()
    created_at = models.DateTimeField(auto_now_add=True)
```

- **PileStatus 状态**：IDLE（空闲） / CHARGING（充电中） / FAULTY（故障） / MAINTENANCE（维护）
- **业务动作**：
  - 安装充电桩（录入信息，状态 IDLE）
  - 启用/停用充电桩
  - 故障报修（IDLE/CHARGING → FAULTY）
  - 维护完成后恢复（FAULTY/MAINTENANCE → IDLE）
  - 计费管理（设置 price_per_kwh）

```python
class TripOrder(models.Model):
    id = models.AutoField(primary_key=True)
    order_no = models.CharField(max_length=50, unique=True)  # 订单编号
    user = models.ForeignKey('User', on_delete=models.CASCADE)  # 乘客
    vehicle = models.ForeignKey('Vehicle', on_delete=models.SET_NULL, null=True)  # 车辆
    start_location = models.CharField(max_length=200)  # 起点
    end_location = models.CharField(max_length=200)    # 终点
    start_lat = models.DecimalField(max_digits=10, decimal_places=7)
    start_lng = models.DecimalField(max_digits=10, decimal_places=7)
    end_lat = models.DecimalField(max_digits=10, decimal_places=7)
    end_lng = models.DecimalField(max_digits=10, decimal_places=7)
    distance = models.DecimalField(max_digits=10, decimal_places=2)  # 距离（km）
    duration = models.IntegerField()  # 时长（分钟）
    fare = models.DecimalField(max_digits=10, decimal_places=2)  # 费用
    status = models.CharField(max_length=20, choices=OrderStatus.choices)
    payment_status = models.CharField(max_length=20, choices=PaymentStatus.choices)
    created_at = models.DateTimeField(auto_now_add=True)
    updated_at = models.DateTimeField(auto_now=True)
```

- **OrderStatus 状态流转**：
  ```
  PENDING → ACCEPTED → IN_PROGRESS → COMPLETED
          ↘ CANCELLED（乘客取消）
  ```
- **PaymentStatus 状态**：UNPAID / PAID / REFUNDED
- **业务动作**：
  - 下单（乘客发起，状态 PENDING）
  - 匹配车辆（PENDING → ACCEPTED，分配车辆和司机）
  - 开始出行（ACCEPTED → IN_PROGRESS）
  - 完成出行（IN_PROGRESS → COMPLETED，计算费用）
  - 支付（PaymentStatus: UNPAID → PAID）
  - 评价（完成后乘客打分和评语）

```python
class MaintenanceRecord(models.Model):
    id = models.AutoField(primary_key=True)
    vehicle = models.ForeignKey('Vehicle', on_delete=models.CASCADE)
    type = models.CharField(max_length=50)          # 维修类型（常规保养/故障维修/事故维修）
    description = models.TextField()                 # 维修描述
    cost = models.DecimalField(max_digits=10, decimal_places=2)  # 维修费用
    start_date = models.DateField()                  # 维修开始日期
    end_date = models.DateField(null=True, blank=True)  # 维修结束日期
    status = models.CharField(max_length=20)         # 进行中/已完成
    created_at = models.DateTimeField(auto_now_add=True)
```

- **关系**：一辆车可以有多条维修记录（一对多）
- **业务动作**：
  - 创建维修记录（自动更新车辆状态为 MAINTENANCE）
  - 完成维修（更新 end_date 和 status，自动恢复车辆状态为 OPERATING）

### 核心页面（≥ 5个）

#### 1. `/map` 地图总览页

- **功能**：使用高德地图展示社区/城市区域全貌，地图上标注所有运营中的车辆实时位置（图标+车牌号）和所有充电桩位置（图标+状态颜色：绿色=空闲，蓝色=充电中，红色=故障）
- **布局**：全屏地图 + 左侧浮动面板（车辆/充电桩数量统计 + 筛选器：按类型、状态筛选）+ 右下角图例
- **交互**：点击车辆图标弹出信息窗（车牌、司机、状态、当前位置）；点击充电桩图标弹出信息窗（编号、类型、功率、状态、价格）
- **消费实体**：Vehicle、ChargingPile
- **API**：`GET /api/vehicles/locations`（获取所有运营车辆位置）、`GET /api/charging-piles/locations`（获取所有充电桩位置）

#### 2. `/vehicles` 车辆管理页

- **功能**：车辆列表管理，支持按类型（VehicleType）、状态（VehicleStatus）筛选；支持注册新车辆、审核车辆、查看详情（含维修记录）、编辑车辆信息
- **布局**：顶部筛选栏 + 车辆数据表格（Ant Design Table 风格，使用 Angular Material Table）+ 行内操作按钮 + 分页
- **二级页面**：`/vehicles/:id` 车辆详情页：基本信息 + 维修记录列表 + 出行记录
- **消费实体**：Vehicle、MaintenanceRecord
- **API**：`GET /api/vehicles`、`POST /api/vehicles`、`PATCH /api/vehicles/:id`、`GET /api/vehicles/:id/maintenance-records`

#### 3. `/charging` 充电桩管理页

- **功能**：左侧地图展示充电桩分布（高德地图），右侧列表展示充电桩详情。支持按类型（PileType）、状态（PileStatus）筛选；支持添加新充电桩、故障报修、维护恢复、修改计费
- **布局**：左侧 60% 高德地图（充电桩标注）+ 右侧 40% 充电桩列表/详情面板
- **交互**：地图标注点击联动右侧详情面板；故障状态的充电桩标注闪烁动画
- **消费实体**：ChargingPile
- **API**：`GET /api/charging-piles`、`POST /api/charging-piles`、`PATCH /api/charging-piles/:id`、`PATCH /api/charging-piles/:id/status`

#### 4. `/orders` 出行订单页

- **功能**：订单列表管理，支持按状态（OrderStatus）、时间范围筛选；支持查看订单详情（含地图轨迹回放）
- **布局**：订单列表表格 + 详情抽屉/弹窗
- **二级页面**：`/orders/:id` 订单详情页：订单信息 + 地图轨迹回放（使用高德地图绘制起终点连线和途经点轨迹）
- **消费实体**：TripOrder
- **API**：`GET /api/orders`、`GET /api/orders/:id`、`PATCH /api/orders/:id/status`

#### 5. `/dashboard` 运营大屏

- **功能**：运营数据概览，包括实时车辆数（在线/总数）、今日订单数、今日收入、充电桩利用率、订单趋势折线图、收入统计柱状图、车辆类型分布饼图
- **布局**：顶部 4 个指标卡片 + 中部 2 个图表（订单趋势 + 收入统计）+ 底部 2 个图表（车辆分布 + 充电桩利用率）
- **消费实体**：Vehicle、TripOrder、ChargingPile
- **API**：`GET /api/dashboard/overview`、`GET /api/dashboard/order-trend`、`GET /api/dashboard/revenue-stats`

### 横切关注点

#### 1. 认证授权 + RBAC

三种角色：**Driver**（司机）、**Operator**（运营人员）、**Admin**（管理员）

| 角色 | 车辆管理 | 充电桩管理 | 订单管理 | 运营大屏 | 数据范围 |
|---|---|---|---|---|---|
| Driver | 查看自己车辆 | 只读 | 查看/更新自己的订单 | 无 | 仅自己的数据 |
| Operator | CRUD | CRUD | CRUD | 只读 | 全部运营数据 |
| Admin | 全部 | 全部 | 全部 | 全部 | 无限制 |

- **后端**：
  - JWT 认证（Django REST Framework 自定义认证类）
  - RBAC 权限类：`IsDriver`、`IsOperator`、`IsAdmin`（DRF Permission 类）
  - 数据范围过滤：司机只能查看自己关联的车辆和订单（通过 queryset filter 实现）
- **前端**：
  - Angular 路由守卫（`AuthGuard`、`RoleGuard`）
  - 根据角色动态生成侧边栏菜单
  - 按钮级权限通过结构型指令 `*appHasRole` 控制
- **触达范围**：DB Users 表 → 后端 DRF 认证类 → 后端 Permission 类 → 后端 queryset filter → 前端路由守卫 → 前端菜单/按钮指令 → 前端 store/service

#### 2. 操作日志

通过 Django 中间件记录关键写操作：

`backend/apps/common/middleware/audit_middleware.py`：
- **记录字段**：操作人（user_id）、HTTP 方法、请求路径、请求体摘要、响应状态码、操作时间、IP 地址
- **必须记录的操作**：
  - 车辆状态变更（审核/维修/报废）
  - 充电桩状态变更（启用/故障/维护）
  - 订单状态流转（下单/接单/完成/取消）
  - 维修记录创建和完成
- **存储**：`audit_logs` 数据库表
- **查询接口**：`GET /api/audit-logs?entity_type=&entity_id=&start_date=&end_date=`

### 全局异常处理

#### 后端统一异常处理

`backend/apps/common/exceptions.py` + `backend/apps/common/middleware/error_handler.py`：
- Django 自定义异常类：`BusinessException`、`ResourceNotFoundException`、`PermissionDeniedException`
- 全局异常处理中间件：捕获所有异常，返回统一 JSON 格式：
  ```json
  {
    "code": 400,
    "message": "业务错误信息",
    "detail": "详细错误（仅开发环境）",
    "timestamp": "2024-01-01T00:00:00Z"
  }
  ```
- DRF 自定义异常处理器：覆盖默认异常响应格式
- 使用 Python logging 记录异常日志

#### 前端请求拦截器统一错误提示

`frontend/src/app/shared/interceptors/error.interceptor.ts`：
- Angular HTTP Interceptor 实现
- 根据 HTTP 状态码统一处理：
  - 401：清除 token，跳转登录页
  - 403：提示"无权限执行此操作"
  - 404：提示"请求的资源不存在"
  - 422：展示表单校验错误
  - 500：提示"服务器内部错误"
- 使用 Angular Material Snackbar 展示错误信息
- 网络超时/断网：提示"网络连接异常，请检查网络"

### 严禁合并职责到单一文件

以下规则必须严格遵守：

1. **model / serializer / view / service 分离**：每个 Django App 必须有独立的 models.py、serializers.py、views.py、services.py 文件，views.py 只负责接收请求和返回响应，业务逻辑在 services.py 中
2. **枚举集中管理**：后端 `common/constants/enums.py` 和前端 `shared/constants/enums.ts` 各维护一份枚举，其他文件 import 使用，不得重复定义
3. **Angular 模块分离**：每个功能模块（vehicles/charging/orders/dashboard/map）独立为 NgModule，支持懒加载，不得将多个模块合并
4. **共用组件独立**：`MapContainer`、`StatusTag`、`StatsCard` 必须在 `shared/components/` 中独立定义，不得内联在页面组件中
5. **拦截器与守卫独立**：`auth.interceptor.ts` 和 `error.interceptor.ts` 各自独立；`auth.guard.ts` 和 `role.guard.ts` 各自独立
6. **异常类独立**：`BusinessException`、`ResourceNotFoundException`、`PermissionDeniedException` 可在同一文件但必须是独立类，不得与业务逻辑混合
7. **服务与组件分离**：Angular Service（如 `map.service.ts`、`api.service.ts`）独立于组件，组件通过依赖注入使用服务
8. **Django 中间件独立**：`audit_middleware.py` 和 `error_handler.py` 各自独立，不得合并

### 核心实体贯穿全栈

每个核心实体必须完整贯穿以下层级：

```
DB Model (apps/*/models.py)
  → Serializer (apps/*/serializers.py)
  → Service (apps/*/services.py)
  → View (apps/*/views.py)
  → API (DRF Router)
    → Angular Model (shared/models/*.model.ts)
      → Angular Service (shared/services/api.service.ts)
        → Component (app/**/*.component.ts)
```

以充电桩 ChargingPile 为例：
```
backend/apps/charging/models.py          # class ChargingPile(models.Model)
backend/apps/charging/serializers.py     # class ChargingPileSerializer
backend/apps/charging/services.py        # class ChargingService
backend/apps/charging/views.py           # class ChargingPileViewSet
frontend/src/app/shared/models/charging-pile.model.ts  # interface ChargingPile
frontend/src/app/shared/services/api.service.ts        # HTTP 请求
frontend/src/app/charging/charging-list/charging-list.component.ts  # 展示列表
```

### 部署要求

项目必须支持 Docker Compose 一键部署，且在 **任意目录名（包括中文目录名）** 下都能正常启动。

docker-compose.yml 要求：
- **不要写 `version:` 字段**
- 顶层声明 `name: ldtransit`
- 所有服务使用 `container_name` 命名容器，带 `${COMPOSE_PROJECT_NAME:-ldtransit}` 前缀
- 数据库数据使用命名卷持久化
- 前端通过 Nginx 反向代理后端 API（调用 `/api`，由 Nginx 转发到 `backend:38503`）
- 数据库配置 `healthcheck`，后端 `depends_on` 数据库健康
- 后端配置 `healthcheck`，前端 `depends_on` 后端健康
- 所有可变配置通过环境变量注入（包括高德地图 API Key: `AMAP_API_KEY`）

需要提供：docker-compose.yml, .env, .env.example, frontend/Dockerfile, backend/Dockerfile, frontend/nginx.conf

### 代码存放规则

这是前后端分离项目。
前端代码放在 `frontend/`
后端代码放在 `backend/`
数据库脚本放在 `database/`
